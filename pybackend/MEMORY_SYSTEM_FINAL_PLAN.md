# Anima 记忆系统最终开发计划（对齐业界实践，增量改造）

## 1. 任务目标
- 在不改现有工具命名体系（`bash/WebSearch/WebFetch/...`）的前提下，补齐可生产的记忆治理能力。
- 记忆能力从“仅文件注入 + 轻量相似度”升级为“可结构化写入、可检索、可追溯、可验证”。
- 保持改动最小，优先复用当前运行时与工具框架。

## 2. 当前假设与边界
- 假设：当前主要后端路径为 `pybackend/anima_backend_core` 与 `pybackend/anima_backend_shared`。
- 假设：记忆持久化先采用工作区内文件存储（`.anima/memory_store.jsonl`），后续可平滑迁移数据库/向量库。
- 假设：本轮先落地“结构化记忆 + 治理策略 + 检索注入 + 测试”，不做图谱记忆与外部向量引擎接入。
- 不确定项：长期是否需要接入外部向量检索服务；本轮先以可替换接口预留迁移空间，不提前引入复杂依赖。

## 3. 成功标准（可验证）
- `memory_add` 写入受策略控制：阈值与证据校验生效，违规写入会被阻止并给出原因。
- `memory_query` 可按 query + topK + 阈值返回稳定结果。
- 系统提示词可自动注入“运行时记忆检索块”，且不影响现有 `user_memory.md` 机制。
- 新增/修改测试全部通过（`python3 -m unittest test_backend_core`）。

## 4. 实施步骤与验收
1. 新增记忆存储与检索模块  
   验证：单测覆盖 add/query 基础行为、排序与阈值。
2. 扩展内建工具（`memory_add`/`memory_query`）并接入治理策略  
   验证：单测覆盖“无证据被拦截”“符合条件可写入”“可检索”。
3. 将运行时检索结果注入 system prompt（自动查询）  
   验证：单测断言 prompt 中出现运行时记忆块。
4. 增加默认配置项（治理开关/阈值）  
   验证：单测与运行路径均能读取默认值，不破坏旧配置。
5. 全量回归测试并记录结果  
   验证：`test_backend_core` 全通过；若失败，修复后再回归。

## 5. 风险与应对
- 风险：文件型记忆在高并发下存在竞争写入风险。  
  应对：本轮保持串行 append + 原子替换读取策略；并在文档中标注后续迁移数据库方案。
- 风险：自动注入记忆导致 prompt 膨胀。  
  应对：严格 `topK` 与文本截断限制。
- 风险：策略过严导致记忆写不进去。  
  应对：默认阈值保守，且返回明确 blocked 原因便于调参。

## 6. 本轮不做
- 不引入外部向量数据库。
- 不引入图谱记忆（Graph Memory）。
- 不重构既有工具命名与调用协议。
