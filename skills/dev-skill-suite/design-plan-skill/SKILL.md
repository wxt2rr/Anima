# 需求到方案（单案直达）

将需求快速收敛为一个可直接开工的方案，强调边界、依据和可验证性。

## 目标

1. 在最短路径内给出可执行方案，而不是发散对比多个版本。
2. 方案必须可验证，且每个关键结论有依据。
3. 默认最小改动，避免超范围设计。

## 适用场景

1. 需求刚提出，需要从“想法”落到“可开发步骤”。
2. 用户强调“按最佳实践直接落地”。
3. 需要在“重构/新增/替换”之间做单次决策。

## 执行流程

1. 明确目标与边界  
   - 写清楚目标、非目标、输入输出、成功标准。  
   - 明确是否允许破旧立新、是否需要兼容旧逻辑。
2. 识别假设与不确定项  
   - 列出“确定事实 / 推断 / 待确认问题”。  
   - 不允许将推断伪装成事实。
3. 给出唯一推荐方案  
   - 只给一个最终方案。  
   - 简述为何不选其他路线（1-2 条关键理由即可）。
4. 拆分最小实施切片  
   - 每步都要能独立验证。  
   - 每步写明影响范围与回滚点。
5. 产出验证矩阵  
   - “改动项 -> 验证命令/检查方式 -> 通过标准”。

## 输出模板

1. 当前理解  
2. 关键假设  
3. 方案结论  
4. 实施步骤（含验证）  
5. 风险与回退  
6. 事实 / 推断 / 建议

## 质量闸门

1. 未定义成功标准，不可进入开发。  
2. 无验证路径的改动，不可进入开发。  
3. 任何“顺手优化”默认拒绝，除非是本次改动直接引发的问题。
